## Security Advisors (`get_advisors`, type=security) — captured live, 2026-07-07

**RLS enabled, no policy (INFO level — effectively default-deny, not an active leak):**
`pin_login_attempts`, `products`, `salon_auth_users`, `salon_device_secrets`, `salon_invites`,
`super_admin_audit_log`. None of these are currently reachable by `anon`/`authenticated` because
Postgres RLS defaults to deny with no matching policy — flagged here because "no policy" is easy
to mistake for "no protection" when read out of context; it's the opposite.

**`SECURITY DEFINER` views (ERROR level):** `public_salon_directory`, `public_rating_lookup`,
`platform_stats`, `public_staff_directory`, `salon_directory`. These run with the view creator's
privileges rather than the querying user's — intentional for these public-facing lookup views,
but worth a deliberate re-confirmation pass, not an assumption, given this is exactly the kind of
issue that caused the `admin_reset_pin`/`subscription_plans` privilege escalations fixed in
migration 011/013.

**Functions with mutable `search_path` (WARN level):** 16 functions, including
`super_admin_update_plan_price`, `verify_pin_for_salon`, `check_pin_lockout`, `verify_staff_pin`,
`update_salon_pin`, `auth_salon_id`, `reactivate_salon`, `suspend_salon`, `validate_invite`,
`consume_invite`, `get_today_staff_stats`, `get_admin_audit_log`, `log_admin_action`,
`super_admin_update_salon`, `super_admin_reset_salon_pin`, `complete_salon_onboarding`,
`create_invite`, `record_subscription_payment`. A mutable `search_path` on a `SECURITY DEFINER`
function is a known Postgres privilege-escalation vector (an attacker-controlled schema earlier
in the path could shadow a table/function the definer relies on). Not confirmed exploitable here,
but the fix (`SET search_path = public, pg_temp` on each function) is cheap and directly
addresses the same class of bug already fixed twice in this codebase.

**`anon`/`authenticated` can execute `SECURITY DEFINER` RPCs (WARN level):** a long list —
`auth_salon_id`, `check_pin_lockout`, `complete_salon_onboarding`, `consume_invite`,
`create_invite`, `get_admin_audit_log`, `log_admin_action`, `public_customer_lookup`,
`reactivate_salon`, `record_subscription_payment`, `salon_is_bookable`,
`super_admin_reset_salon_pin`, `super_admin_update_plan_price`, `super_admin_update_salon`,
`suspend_salon`, `update_salon_pin`, `validate_invite`, `verify_pin_for_salon`,
`verify_staff_pin`. Most of these are *supposed* to be anon-callable (public booking page,
staff PIN login before a session exists) — but `get_admin_audit_log`, `super_admin_update_salon`,
`suspend_salon`, `reactivate_salon`, `super_admin_update_plan_price`, and
`super_admin_reset_salon_pin` being anon-executable is worth a direct look: these are named as
super-admin-only actions, and their actual protection (if any) must be inside the function body
via an `app_metadata` check, not at the grant level. **This needs verification, not assumption** —
flagged for a follow-up, out of scope of Steps 1–5 but adjacent to the tenancy work.

**Auth-level:** leaked password protection (HaveIBeenPwned check) is currently disabled for
Supabase Auth. Affects Super Admin / Sales Rep email+password logins, not the PIN-based staff
flow. Low effort to enable, unrelated to tenancy work.

## Orphaned migration file

`supabase/migrations/015_allow_anon_payment_status_update.sql` exists in the repo alongside
`015_add_bookings_anon_update_payment_status.sql`. Confirmed via live schema inspection: the
database's actual `payment_status` transition logic matches `016_staff_confirmed_payments_and_mpesa_columns.sql`
(the `awaiting_confirmation` model), not the direct `pending → paid_upfront` logic the orphaned
file proposes. **Decision needed** (flagged in the plan, not resolved here): keep the orphaned
file as a documented historical artifact, or delete it now that it's confirmed superseded.
